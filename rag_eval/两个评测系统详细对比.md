# 两个评测系统详细对比

## 一、原项目检索配置分析

### 1.1 文本分块策略（高级）

原项目使用**三层分块策略**，比 rag_eval 的简单分块更智能：

```python
# splitters/splitter_md.py
text_chunk_size = 1000  # 默认值

# 第一层：按 Markdown 标题分割
text_splitter = MarkdownHeaderTextSplitter(headers_to_split_on=[
    ("#", "Header 1"),
    ("##", "Header 2"),
    ("###", "Header 3"),
])

# 第二层：语义分割（超过 1000 字符的块）
semantic_splitter = SemanticChunker(embedding, breakpoint_threshold_type="percentile")

# 第三层：图片单独处理
```

| 分块方式                   | 说明                     | 适用场景       |
| -------------------------- | ------------------------ | -------------- |
| MarkdownHeaderTextSplitter | 按标题层级分割           | 保留文档结构   |
| SemanticChunker            | 语义分割                 | 长段落智能切分 |
| text_chunk_size=1000       | 超过 1000 字符才语义分割 | 避免过度分割   |

**为什么这样设计**：

1. **保留结构**：按标题分割保留文档层级关系
2. **语义完整**：语义分割在语义边界切分，不会切断句子
3. **图片独立**：图片单独处理，支持多模态检索

**与 rag_eval 对比**：

| 对比项   | 原项目           | rag_eval         |
| -------- | ---------------- | ---------------- |
| 分块方式 | 语义分割         | 固定字符分割     |
| 分块大小 | 动态（语义决定） | 固定（512/1024） |
| 结构保留 | ✅ 按标题分割    | ❌ 无            |
| 智能程度 | 高               | 低               |

### 1.2 向量模型配置

| 配置项   | 值                           | 说明                       |
| -------- | ---------------------------- | -------------------------- |
| 嵌入模型 | `Qwen/Qwen3-VL-Embedding-8B` | 多模态模型，支持文本+图片  |
| 向量维度 | 4096                         | 高维度，语义表达能力强     |
| 度量方式 | IP (内积)                    | 适合归一化向量，计算相似度 |

**为什么选择这个模型**：

- 支持多模态（文本+图片），符合项目"Multimodal RAG"定位
- 4096 维度在中文场景下效果好
- SiliconFlow API 托管，无需本地部署

### 1.2 索引配置

```python
# 稀疏向量 (BM25)
index_type = "SPARSE_INVERTED_INDEX"
metric_type = "BM25"
bm25_k1 = 1.2    # 词频饱和度
bm25_b = 0.75    # 文档长度归一化
inverted_index_algo = "DAAT_MAXSCORE"

# 密集向量
index_type = "AUTOINDEX"
metric_type = "IP"
nprobe = 10
```

**参数解释**：

| 参数            | 值           | 含义                                            |
| --------------- | ------------ | ----------------------------------------------- |
| `bm25_k1 = 1.2` | 低饱和度     | 一个词出现 3 次和 10 次差别不大，避免高频词主导 |
| `bm25_b = 0.75` | 中等惩罚     | 长文档会被适度惩罚，避免"长篇大论"占优势        |
| `nprobe = 10`   | 中等搜索范围 | 在 IVF 索引中搜索 10 个聚类，平衡速度和精度     |

### 1.3 检索策略

```python
# 默认检索方式
results = m_re.hybrid_search(embedding, query, limit=3)
```

| 配置项     | 值        | 说明           |
| ---------- | --------- | -------------- |
| 检索方式   | 混合检索  | 向量 + BM25    |
| 向量权重   | 1.0       | 语义检索权重   |
| 关键词权重 | 1.0       | 关键词检索权重 |
| 返回数量   | top_k = 3 | 返回 3 个结果  |

**为什么这样设置**：

1. **混合检索**：中文技术文档需要关键词精确匹配（如"Flink"、"Watermark"），纯向量可能漏掉
2. **等权重 (1.0, 1.0)**：默认值，不偏向任何一种检索方式
3. **top_k = 3**：平衡上下文完整性和 LLM 上下文窗口限制

### 1.4 特殊处理

```python
# 图片检索：只能用向量检索
if os.path.isfile(query):
    results = self.dense_search(embedding, limit=self.top_k)
else:
    results = self.hybrid_search(embedding, query, limit=self.top_k)
```

**原因**：图片无法做 BM25 全文检索，只能用向量检索

| 配置            | 效果             | 适用场景         |
| --------------- | ---------------- | ---------------- |
| chunk_size=256  | 小块，检索精准   | 短问答、FAQ      |
| chunk_size=512  | 中等，平衡       | **推荐默认值**   |
| chunk_size=1024 | 大块，上下文完整 | 长文档、技术说明 |

**你的项目适合**：512（中文技术文档，段落长度适中）

#### BM25 关键词检索

| 配置           | 效果                       |
| -------------- | -------------------------- |
| use_bm25=False | 纯向量检索，依赖语义相似度 |
| use_bm25=True  | 混合检索，关键词 + 语义    |

**你的项目适合**：True（Flink 术语需要精确匹配）

#### HyDE (假设文档嵌入)

| 步骤              | 说明                       |
| ----------------- | -------------------------- |
| 1. 生成假设答案   | LLM 先生成一个"可能的答案" |
| 2. 用假设答案检索 | 用假设答案的向量去检索     |

**效果**：提升查询与文档的语义匹配度

**你的项目适合**：可以尝试（技术问题的假设答案通常比较准确）

#### Rerank (重排序)

| 步骤        | 说明                           |
| ----------- | ------------------------------ |
| 1. 初始检索 | 检索 initial_top_k=50 个候选   |
| 2. 精排     | 用 BGE reranker 精排到 top_k=3 |

**效果**：提升最终结果的相关性

**你的项目适合**：可以尝试（reranker 对中文效果好）

#### MultiQuery (多查询)

| 步骤        | 说明                  |
| ----------- | --------------------- |
| 1. 改写问题 | LLM 生成 3 个不同问法 |
| 2. 多次检索 | 每个问法都检索一次    |
| 3. 合并结果 | 去重并排序            |

**效果**：覆盖不同表达方式，减少遗漏

**你的项目适合**：可以尝试（用户问法可能与文档表述不同）

#### Parent-Child (父子分块)

| 步骤                  | 说明                      |
| --------------------- | ------------------------- |
| 1. 大块 (parent)      | 1024 字符，保留完整上下文 |
| 2. 小块 (child)       | 512 字符，检索精准        |
| 3. 检索小块，返回大块 | 兼顾精准和完整            |

**效果**：检索更精准，同时保留上下文

**你的项目适合**：可以尝试（技术文档有层次结构）

## 四、为什么这样配置？

### 4.1 原项目配置设计考量

**为什么选择混合检索？**

1. **简单优先**：默认配置够用，不需要复杂调参
2. **稳定优先**：混合检索是成熟方案，效果有保障
3. **中文适配**：BM25 + jieba 分词对中文技术术语（如"Flink"、"Watermark"）精确匹配效果好
4. **多模态支持**：图片只能用向量检索，文本用混合检索，统一接口

**为什么用等权重 (1.0, 1.0)？**

- 不偏向任何一种检索方式
- 让向量和关键词同等重要
- 简单有效，避免调参

**为什么 top_k=3？**

- 平衡上下文完整性和 LLM 上下文窗口
- 太多会稀释相关信息，太少会遗漏
- 3 是经验值，适合大多数场景

### 4.2 rag_eval 多配置系统设计考量

**为什么测试这 4 个配置？**

| 配置                         | 测试目标     | 设计理由                     |
| ---------------------------- | ------------ | ---------------------------- |
| 配置 1: 纯向量               | 基准线       | 测试 BM25 是否有必要         |
| 配置 2: 混合检索             | 与原项目对比 | 验证原配置是否最优           |
| 配置 3: 混合 + HyDE          | 查询扩展     | 测试 HyDE 是否提升语义匹配   |
| 配置 4: 混合 + HyDE + Rerank | 精排         | 测试 Rerank 是否提升排序质量 |

**为什么逐步叠加？**

```
配置 1 → 配置 2：测试 BM25 的影响
配置 2 → 配置 3：测试 HyDE 的影响
配置 3 → 配置 4：测试 Rerank 的影响
```

**这样设计的好处**：

1. **控制变量**：每次只改一个参数，知道哪个起了作用
2. **逐步优化**：从简单到复杂，逐步提升
3. **成本可控**：先测试免费的，再测试付费的

**为什么测试 overlap 和 rrf_k？**

| 参数    | 测试理由                               |
| ------- | -------------------------------------- |
| overlap | 影响边界信息保留，可能提升上下文完整性 |
| rrf_k   | 影响排名融合权重，可能优化检索结果排序 |

### 4.3 各配置预期效果

| 配置                | 预期效果                 | 成本             |
| ------------------- | ------------------------ | ---------------- |
| 配置 1: 纯向量      | 基准，可能漏掉关键词匹配 | 最低             |
| 配置 2: 混合检索    | 提升关键词匹配           | +BM25 计算       |
| 配置 3: +HyDE       | 提升语义匹配             | +1 次 LLM 调用   |

## 五、推荐评测方案

### 5.1 设计思路：为什么这样搭配配置？

**核心原则：逐步叠加，控制变量**

这 4 个配置是**逐步叠加**的设计，每次只改变一个参数，目的是清楚地知道哪个策略起了作用：

```
配置 1（纯向量）
    ↓ +BM25
配置 2（混合检索）
    ↓ +HyDE
配置 3（混合 + HyDE）
    ↓ +Rerank
配置 4（混合 + HyDE + Rerank）
```

| 对比        | 测试目标        | 结论                   |
| ----------- | --------------- | ---------------------- |
| 配置 1 vs 2 | BM25 有用吗？   | 知道是否需要关键词检索 |
| 配置 2 vs 3 | HyDE 有用吗？   | 知道是否需要查询扩展   |
| 配置 3 vs 4 | Rerank 有用吗？ | 知道是否需要精排       |

**可以其他排列组合吗？**

完全可以！比如：

| 配置              | 组合    | 测试目标                       |
| ----------------- | ------- | ------------------------------ |
| 混合 + Rerank     | ✅ 可以 | 测试没有 HyDE 时 Rerank 的效果 |
| 纯向量 + Rerank   | ✅ 可以 | 测试纯向量 + Rerank 的效果     |
| 纯向量 + HyDE     | ✅ 可以 | 测试 HyDE 对纯向量的提升       |
| 混合 + MultiQuery | ✅ 可以 | 测试多查询的效果               |

**为什么没加这些？**

1. **配置太多会很慢**：5 个配置已经需要很久
2. **逐步测试更清晰**：先测基础的，再测高级的
3. **避免混淆**：太多变量一起变，不知道哪个起了作用

### 5.5 参数调优实验

除了检索策略，还可以测试以下参数：

#### 测试 Overlap 参数

**Overlap（重叠大小）** 控制相邻文本块之间的重叠字符数，作用是防止信息在块边界处丢失。

```
┌─────────────────────────────────────────────────────────────┐
│  Overlap 原理                                                │
│  ─────────────────────────────────────────────────────────── │
│  无重叠：[块1: 0-512] [块2: 512-1024]                        │
│                      ↑                                      │
│                 边界信息可能丢失                              │
│                                                             │
│  有重叠：[块1: 0-512] [块2: 462-974]                         │
│                   ↑↑↑↑↑                                     │
│                重叠 50 字符                                  │
│                保留边界上下文                                 │
└─────────────────────────────────────────────────────────────┘
```

| overlap 值 | 效果                 | 适用场景               |
| ---------- | -------------------- | ---------------------- |
| 0          | 无重叠，最节省空间   | 文档结构清晰，段落独立 |
| 64         | 少量重叠（约 12.5%） | **推荐默认值**         |
| 128        | 中等重叠（约 25%）   | 长段落，上下文依赖强   |
| 200        | 大量重叠（约 40%）   | 极长文档，上下文强依赖 |

#### 测试 RRF-K 参数

**RRF-K（Reciprocal Rank Fusion 常数）** 用于混合检索中融合向量检索和 BM25 检索的排名。

```
RRF_score(d) = Σ 1 / (k + rank_i(d))

其中：
- d = 文档
- k = RRF 常数（默认 60）
- rank_i(d) = 文档 d 在第 i 种检索方式中的排名
```

| rrf_k 值 | 效果                   | 适用场景           |
| -------- | ---------------------- | ------------------ |
| 30       | 排名靠前的文档权重更高 | 检索结果质量差异大 |
| 60       | 平衡权重（推荐默认）   | 通用场景           |
| 100      | 排名权重更平滑         | 检索结果质量接近   |

**示例**：

```
文档 A：向量排名 #1，BM25 排名 #5
文档 B：向量排名 #3，BM25 排名 #2

rrf_k=60 时：
文档 A 得分 = 1/(60+1) + 1/(60+5) = 0.0164 + 0.0154 = 0.0318
文档 B 得分 = 1/(60+3) + 1/(60+2) = 0.0159 + 0.0161 = 0.0320

→ 文档 B 略胜（因为两项排名都较靠前）
```

**参数调优实验方法**：

1. 先用第一批配置找出最优检索策略
2. 保持最优策略不变，只改变一个参数
3. 对比 Elo 评分，找出最优参数值
